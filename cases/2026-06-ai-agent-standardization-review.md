# AI Agent 项目交付复盘：从定制交付到标准化产品平台

## 案例摘要

本项目是百瑞围绕 AI Agent 能力进行的一次真实客户交付和需求共创。项目过程中，我们与客户进行了多轮线下沟通，共同梳理 Agent 使用场景、GitHub 项目读取、消息触达、定时任务、微信连接、热点信息、模型 API 配置和部署上线等需求。

这单的核心收获不是某个单点功能，而是暴露出一个更关键的问题：如果没有平台化产品、标准化配置流程和交付冻结机制，在产品真正成型前，客户会不断改变方向、补充功能、调整技术路径，研发和部署会被迫反复返工。最终，技术交付会变得痛苦，客户也很难判断“什么算完成”。

因此，本项目复盘结论是：百瑞后续不能继续以纯定制项目方式交付 Agent，而要建设标准化网站平台，将模型、GitHub、微信、知识库、定时任务、角色提示词、部署状态、验收清单等能力产品化。客户在平台上按标准流程配置和确认需求；在产品生成或部署前，需求必须冻结，变更必须走版本和评审机制。

## 业务背景

客户希望获得一个可商用的 AI Agent 产品能力，重点包括：

- 通过网页完成模型 API、GitHub 应用、微信连接等设置。
- Agent 能读取 GitHub 上的公开项目和客户自己的项目库。
- Agent 能主动发送消息，进行教学、提醒、复习、监督和行为反馈。
- Agent 有明确角色设定和持续记忆，而不是普通聊天机器人。
- 前端品牌必须使用公司品牌，不暴露不适合商用的历史项目字段或 Logo。
- 部署到公网域名后可直接使用，后续能继续扩展后台、报表和多任务。

项目关联的只读工程仓库：

- [LUTAO581314/赫尔墨斯-](https://github.com/LUTAO581314/hermes-)：百瑞 Agent OS / Hermes 运行时方向，沉淀 Agent 内核、记忆、连接器、前端契约、部署等能力。
- [LUTAO581314/MOXI-cloud-agent](https://github.com/LUTAO581314/MOXI-cloud-agent)：百瑞商业平台方向，负责官网、客户控制台、部署向导、授权、服务器管理、验收和支持流程。

## 线下与客户共创出的需求

本项目中，需求不是一次性明确下来的，而是在不断试用、部署和反馈中逐步显形。客户共创出的重点需求包括：

- 品牌统一：前端、字段、Logo、标题、设置面板不能暴露非公司品牌。
- 模型配置：网页登录后弹出设置窗口，配置模型 API、中转站 Base URL、模型名称和密钥。
- GitHub 读取：预留 GitHub 应用配置，支持读取公开仓库和客户自有仓库。
- 微信连接：希望通过扫码或适配层连接微信，实现消息接收和推送。
- 主动教学：Agent 不等用户提问，能主动讲项目知识、命令、考公、入党材料等。
- 定时任务：支持艾宾浩斯复习、每日任务、定时抽查和提醒。
- 角色提示词：核心人格设定必须可配置、可持久化，并在服务重启后继续生效。
- 热点面板：前端要有热点信息入口，能支持外部信息和热点任务。
- 部署可用：域名解析、内存、后台服务、带宽、日志、服务状态都需要可检查。
- 验收确认：部署后需要让用户确认哪些功能已实现、哪些还缺外部配置。

这些需求本身都有价值，但它们在非标准化交付中不断插入，会造成技术路径频繁变化。

## 遇到的问题

### 1. 需求边界不冻结，方向持续变化

项目开始时，需求集中在“部署一个 Agent”。随着体验推进，需求逐步扩展到 GitHub、微信、定时任务、热点面板、角色人格、模型中转站、商用品牌、后端能力验证等。

如果没有明确的需求冻结点，客户每发现一个新想法，就会要求立刻进入当前版本。研发侧会不断打断原本的部署和验证节奏，导致功能表面变多，但稳定性、验收标准和交付节奏变差。

### 2. 产品标准化不足，配置都变成手工部署

模型 API、Base URL、GitHub 应用、微信连接、TTS、域名、服务端口、系统服务、日志排查等都需要人工逐项处理。没有统一的平台配置向导时，部署人员需要直接改配置文件、查日志、重启服务、排查前端报错。

这会导致交付强依赖工程师经验，客户无法自助完成，也无法清楚知道当前卡点是“配置没填”“渠道不可用”“服务没启动”还是“代码错误”。

### 3. 客户把产品试用反馈和研发变更混在一起

客户在试用过程中会提出大量即时反馈，例如界面是不是历史原型界面、Logo 是否干净、二维码如何出、输入为什么没输出、TTS 为什么报错、热点按钮在哪里等。

这些反馈有些是产品体验问题，有些是外部服务配置问题，有些是部署问题，有些是研发缺陷。如果没有标准化问题分类和验收清单，所有反馈都会变成“马上修”，团队会陷入高压救火。

### 4. 第三方能力不稳定会被误判为产品失败

项目中模型中转站 Key 可以认证，但中转站后台没有给 default 分组配置可用模型渠道，导致请求返回“没有可用渠道”。从客户视角看，这是“Agent 没回复”；从技术视角看，后端接收、入库、服务运行都是正常的，失败点在第三方渠道配置。

如果平台没有把“模型 Key 是否有效”“模型名是否可用”“渠道是否启用”“分组是否匹配”做成清晰检测，客户会把外部配置问题误认为产品不可用。

### 5. 商用品牌和开源来源边界需要严格治理

本项目要求前端只出现公司品牌，不暴露历史项目名、上游项目名、旧 Logo 或不适合商用的字段。这说明商业产品不能只靠替换几个标题，而需要建立品牌字段清单、扫描规则和发布前检查。

同时，工程上可以参考成熟开源项目，但客户可见界面、合同、配置、激活流程和文案必须统一为百瑞品牌。

## 关键复盘

### 需求层复盘

客户不是一开始就能完整描述自己想要的 Agent。很多需求是在看到系统跑起来后才说清楚的。这是 AI Agent 类产品的常态。

因此，百瑞不能期待客户一次性给出完整 PRD，而应该提供标准场景模板：

- 学习监督 Agent。
- GitHub 项目学习 Agent。
- 客户服务 Agent。
- 企业知识库 Agent。
- 销售和运营 Agent。
- 私有部署助理 Agent。

客户先选场景，再填配置，而不是让客户自由描述一切。

### 产品层复盘

本项目说明，Agent 产品必须把复杂能力拆成可配置模块：

- 模型配置模块。
- GitHub 连接模块。
- 微信和消息通道模块。
- 角色提示词模块。
- 定时任务模块。
- 知识库和记忆模块。
- 热点信息模块。
- 部署状态和健康检查模块。
- 验收清单模块。

这些模块应该在网站平台里呈现为表单、状态、测试按钮和错误说明，而不是让工程师在服务器上手动排查。

### 技术层复盘

技术路径应从“一台服务器手工部署一个 Agent”升级为“平台生成标准配置并交付运行时”：

```text
客户选择场景
  -> 平台收集模型 / GitHub / 微信 / 角色 / 定时任务配置
  -> 生成配置草稿
  -> 自动检测缺失项和外部连通性
  -> 客户确认需求冻结
  -> 平台生成部署包或远程部署任务
  -> 服务启动
  -> 健康检查
  -> 客户验收
  -> 后续变更进入版本管理
```

该路径与两个关联仓库的职责边界一致：

- Hermes 负责 Agent OS 内核、记忆、工具、连接器、运行时和前端契约。
- MOXI Cloud Agent 负责网站、客户控制台、部署向导、授权、服务器管理、验收报告和支持流程。

### 交付层复盘

部署痛苦的本质不是某条命令难，而是交付过程缺少产品化抽象。工程师一边改代码、一边改配置、一边查第三方服务、一边解释需求边界，必然会越来越痛苦。

后续应把交付拆成：

- 售前需求确认。
- 标准场景选择。
- 配置采集。
- 外部账号和渠道检测。
- 需求冻结。
- 部署执行。
- 健康检查。
- 验收确认。
- 变更管理。

每一步都要有页面、有状态、有日志、有负责人。

## 标准化产品建议

### 1. 建设百瑞 Agent 平台，而不是继续做零散项目

平台应提供统一入口：

- 官网展示产品能力和案例。
- 客户登录后创建 Agent。
- 选择场景模板。
- 配置模型、GitHub、微信、知识库、角色和定时任务。
- 查看部署状态、日志、健康检查和验收结果。
- 后续变更通过版本草稿提交。

### 2. 产品做出来前，需求不能随意改

建议建立“需求冻结确认”机制：

- 客户在平台上确认当前版本包含哪些功能。
- 冻结后进入开发或部署。
- 新想法进入下一版本需求池。
- 紧急变更必须说明影响、成本和延期风险。

这样可以保护研发节奏，也让客户知道每次变化都不是免费的隐形成本。

### 3. 把配置测试做成平台能力

平台至少要支持：

- 模型 Key 测试。
- Base URL 自动补全和校验。
- 模型名可用性测试。
- GitHub App / Token 权限测试。
- 微信连接状态测试。
- TTS 凭证测试。
- 域名和服务健康检查。
- 后台服务状态和日志摘要。

测试失败时，平台要告诉客户是哪里失败，而不是只显示“Agent 没回复”。

### 4. 建立品牌和商用发布检查

每次发布前检查：

- 页面标题。
- Logo。
- 按钮和菜单文字。
- 配置字段。
- README 和公开文档。
- 前端打包产物。
- 错误提示。
- 第三方项目名是否泄露到客户界面。

内部可以保留工程来源和开源许可证说明，但客户可见界面必须统一为百瑞品牌。

### 5. 把案例库变成产品资产

本案例后续应继续补充：

- 客户行业和规模。
- 具体使用场景。
- 部署前后对比。
- 沟通纪要。
- 需求变更记录。
- 验收截图。
- 故障复盘。
- 可复用配置模板。

每一单都要反哺平台，而不是只留在聊天记录和工程师记忆里。

## 可复用结论

这单证明，百瑞 AI Agent 的价值方向是成立的：客户确实需要一个能连接 GitHub、微信、模型、定时任务、角色人格和业务知识的主动型 Agent。

但这单也证明，如果没有平台化和标准化，交付会快速失控。客户会不断提出新方向，研发会不断改技术路径，部署会变成长期救火。

下一阶段的核心不是继续堆功能，而是把功能变成平台里的标准模块、配置流程、检测能力和验收机制。只有这样，百瑞才能从“做一单算一单”升级为“卖可复制的 Agent 产品”。

## 下一步行动

- 在 MOXI Cloud Agent 中优先建设客户配置控制台和部署向导。
- 在 Hermes 中明确运行时配置契约，减少手工改配置。
- 建立需求冻结、版本变更和验收清单。
- 为 GitHub、模型中转站、微信、TTS 增加标准连通性检测。
- 每个客户项目结束后，按本案例模板补充案例库。
